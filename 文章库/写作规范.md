## 一、写作前的准备

### 1.0 【核心流程】剧情走向选择

**两种模式：深度写作 vs 快速写作**

---

#### 【深度写作】完整流程（重要章节）

**第1轮：章节核心走向**

**输出：**
1. **本章概述**：核心事件、起承转合（约150字）
2. **4个选项**：A/B/C/D 四个不同的剧情走向

**选项设计要求：**
- 每个选项必须有**差异化**，情感基调/剧情方向明显不同
- **核心选项**（最符合剧情逻辑的）标注★，并写明理由
- 其他选项简略描述，不每个都详细分析

**示例格式：**
```
A. 【紧张型】林远舟决定立刻出发 | ★核心选项，符合角色责任感
B. 【冲突型】林远舟犹豫是否告诉妻子真相
C. 【悬疑型】总部隐瞒了关于源点协议的某些信息
D. 【温情型】出发前与妻子的告别场景
```

**用户选择：** 选A/B/C/D 或 提出修改意见

---

**第2轮：完整框架确认**

**基于第1轮选择，输出：**
1. **本章完整框架**：
   - 开篇：如何切入
   - 发展：核心事件展开
   - 高潮：情感/冲突峰值
   - 结尾钩子：具体设计
2. **写作要点提醒**：
   - 必须包含的感官描写
   - 必须回收的伏笔
   - 风格融合提醒

**用户确认：** OK/修改/补充

---

**→ 确认后直接开始写作**

---

#### 【快速写作】精简流程（日常更新）

**触发条件：** 用户说「快速写」或章节较为简单

**流程：**
1. 读剧情摘要
2. 确认核心走向 + 特殊要求
3. 直接写作
4. 写完自检 + 复查
5. 更新摘要

---

#### 【续写】保持节奏

**触发条件：** 用户说「续写」或「继续」

**流程：**
1. 读上文结尾3段
2. 直接写
3. 复查 + 更新摘要

## 二、章节规划

### 2.1 章节结构模板

```
EP0X-XX：章节名
【时间/地点标记】

[第一段：具象化感官场景，建立当下危机]
 ↓
[第二段：补充背景信息（倒叙/对话）]
 ↓
[第三段：核心冲突显现]
 ↓
[第四段：选择与代价]
 ↓
[第五段：新发现/新谜题]
 ↓
【钩子结尾：未完成句子/突然打断/危险暗示】
【本章完】
```

### 2.2 单章字数控制

| 章节类型 | 字数范围 |
|----------|----------|
| 标准章节 | 1500-1800 字 |
| 重要转折章节 | 1800-2200 字 |
| 过渡章节 | 1000-1500 字 |

### 2.3 每章必须包含的元素

| 元素 | 说明 | 风格来源 |
|------|------|----------|
| **感官描写** | 视觉/听觉/触觉/嗅觉，至少2种 | Reynolds+刘慈欣 |
| **一个谜题** | 角色或读者想知道答案的问题 | Asimov |
| **一个代价** | 科技进步/选择带来的后果 | 刘慈欣 |
| **被观测感** | 暗示某种存在在观察 | Clarke |
| **钩子结尾** | 未完成句子或危险暗示 | Herbert |

## 三、写作进行时

### 3.1 开篇写法

**原则**：直接扔进悬念场景，不解释背景

```
❌ 错误："林远舟是一名量子科学家，他的工作是……"
✓ 正确："林远舟从噩梦中惊醒。"
```

**三种开篇方式**：

1. **感官冲击型**：从危机/异常开始
   ```
   身体化作无数碎片，每一片都在尖叫。
   ```

2. **对话型**：从一句奇怪的话开始
   ```
   "找到你了。"
   ```

3. **氛围型**：从环境描写开始
   ```
   舷窗外，地球悬浮在虚空中，像一颗被薄云包裹的蓝色弹珠。
   ```

### 3.2 对话写法

**原则**：对话中自然融入世界观设定

```
❌ 错误：
  "我是联邦的量子科学家，专门研究意识上传技术。"
  "意识上传是我们五十年来的伟大发明。"

✓ 正确：
  "距离上次意识上传已过去17小时34分钟。"
  系统没有回答他的问题，只是继续播报着那串冰冷的数字。
```

### 3.3 具象化感官描写

**规则**：每章至少3处具象化描写

| 场景 | 感官描写示例 |
|------|-------------|
| 坠落噩梦 | 被细丝般的冰锥割裂皮肤、融化与凝固并存 |
| 芯片发热 | 像烙铁一样烫、有东西在通过它向外传输 |
| 源点协议 | 低频嗡鸣、低频震动，光线扭曲 |
| 织梦者显现 | 无数双眼睛同时注视、穿透皮肤直抵骨髓 |

### 3.4 时间/地点标记

使用统一格式：

```
【时间】 | 【地点】
例如：
2075年3月15日 03:47 | 地球轨道观测站「拉格朗日-3」

当晚 21:15 | 林远舟私人舱室
```

## 四、配图生成流程

### 4.1 图片规格（必须严格遵守）

| 参数 | 值 |
|------|-----|
| **分辨率** | **1280×720** |
| **宽高比** | 16:9 横幅 |
| **格式** | JPG |
| **存放路径** | `E:\Opencode\科幻小说\images\` |
| **命名规则** | `ep{集}{章}.jpg`（如 `ep0103.jpg`） |

**⚠️ 生成后必须用 PIL 检查分辨率，不符合必须重生成！**

### 4.2 生成时机

| 阶段 | 时机 |
|------|------|
| 配图 | 本章内容写完后、发布前 |
| 封面 | 项目启动时/重要更新时 |

### 4.3 提示词模板

```
科学家手绘笔记风格，铅笔线条画，[具体场景描述，与本章内容高度匹配]，
手写英文字母涂鸦边缘，潦草的物理公式和图表，yellowed paper texture，
细腻铅笔阴影，白底，无黑边，无边框，16:9 cinematic
```

### 4.4 生成命令

```python
image_generate(
    prompt="...",
    model="minimax/image-01",
    size="1280x720"
)
# 生成后用 PIL 检查：assert img.size == (1280, 720)
```

### 4.5 生成后处理

```
1. 用 PIL 检查分辨率（必须为 1280×720）
2. 保存到 E:\Opencode\科幻小说\images\
   命名：ep0103.jpg（第3章为 ep0103，第10章为 ep0110）
3. 用 Start-Process 打开图片让用户查看
4. git add + commit + push
```

## 五、写作后的流程

### 5.1 内容输出

1. **保存章节文件**
   ```
   路径：E:\Opencode\科幻小说\chapters\EP01-03.md
   格式：Markdown，纯文字内容
   ```

2. **更新阅读器 index.html**
   ```
   文件：E:\Opencode\科幻小说\index.html
   更新：CHAPTER_ORDER 数组末尾添加 "chapters/EP01-03.md"
   （IMAGE_MAP 无需手动维护，getImagePath() 自动推导）
   ```

3. **生成并保存配图**
   ```
   - 分辨率：1280×720（16:9）
   - 存放：E:\Opencode\科幻小说\images\ep0103.jpg
   - 命令：image_generate(model="minimax/image-01", size="1280x720")
   - 检查：用 PIL 确认 img.size == (1280, 720)
   - 查看：Start-Process 打开图片
   - 提交：git add + commit + push
   ```

4. **【重要】更新剧情摘要**
   ```
   文件：E:\Opencode\科幻小说\文章库\剧情摘要.md
   更新内容：
   - 新增本章摘要（约100-200字）
   - 更新角色状态
   - 记录新出现的谜题/伏笔
   - 标记需要回收的线索
   ```

5. **更新记忆文件**
   ```
   文件：E:\Opencode\科幻小说\文章库\小说记忆文件.md
   更新：当前进度标记
   ```

### 5.2 复查审核

**融入写作过程：每写完一段→对照检查→问题立即修改**

#### 写作中自检

```
每写完一段时快速检查：
□ 这段是否推进剧情？
□ 感官描写够不够？
□ 有没有"被观测"暗示？
```

#### 写完后整体复查

```
【必须朗读检查】
□ 语句是否通顺？朗读检查，避免语病
□ 有无重复表达？（句式/词汇/意象）
□ 段落之间逻辑是否连贯？

□ 角色性格是否符合设定？
□ 时间线是否正确（2075年）？
□ 科技设定是否一致？
□ 开篇是否直接进入场景？
□ 是否有具象化感官描写（至少3处）？
□ 钩子结尾是否够强？
□ 字数是否在1500-1800字范围？
□ 配图场景是否匹配？
□ 风格融合是否到位？
□ 是否回收了之前的伏笔？
```

**⚠️ 复查时必须朗读全文，确保语句通顺！**

#### 复查处理

```
发现问题 → 立即修改 → 再次检查
                ↓
            无问题 → 发布
```
